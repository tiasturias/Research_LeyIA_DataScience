## 1. Diagnóstico metodológico que motiva el cambio

### 1.1 Problema central

La Fase 5 actual todavía arrastra artefactos de una lógica predictiva heredada. El output actual contiene una columna `split`, genera o espera archivos tipo `mvp_train_test_split.csv` y `phase6_train_test_split.csv`, y conserva narrativa documental de `train/test` en el workbook auditable.

Esto es incompatible con la pregunta real del proyecto:

> ¿Existe una asociación estadísticamente significativa entre las características de la regulación de IA de un país y el desarrollo de su ecosistema de IA, después de controlar por factores socioeconómicos e institucionales?

Esa pregunta es inferencial, comparativa y observacional. No es una pregunta de predicción externa. Por tanto, con una muestra de 43 países, separar 9 países como test set produce más daño que beneficio.

### 1.2 Por qué el `train/test split` debe eliminarse

Con N=43, un split 34/9 tiene cuatro problemas severos:

| Problema | Impacto metodológico |
|---|---|
| Reduce el N útil | Excluye alrededor de 20% de países de la estimación principal, justo cuando la muestra ya es pequeña. |
| No es validación externa real | Un split aleatorio dentro del mismo dataset es validación interna débil, no evidencia de generalización externa. |
| Puede sesgar geopolíticamente el holdout | En el Excel actual, el supuesto test set queda concentrado en Europa y LATAM, sin representación de varias regiones clave. |
| Confunde auditoría y relato político | Si Fase 6 usa todos los países, el test set deja de ser test; si no los usa, se pierde potencia. |

La solución correcta es eliminar el split y reemplazarlo por una tabla de membresía analítica (`analysis_sample_membership.csv`). Esa tabla no divide países para entrenamiento/prueba; documenta quién pertenece a la muestra primaria y qué flags de sensibilidad se usarán luego en Fase 7.

### 1.3 Naturaleza correcta de Fase 5

La Fase 5 correcta debe definirse así:

> Fase 5 es una fase de preparación auditable y contractual de datos, no una fase de pre-modelado predictivo.

Su función es preparar un paquete confiable para Fase 6, no generar métricas predictivas ni reservar un holdout.

### 1.4 Fundamento bibliográfico resumido

El cambio se sostiene en mejores prácticas de estadística aplicada y reporte de estudios observacionales:

- **TRIPOD / TRIPOD-Cluster:** un random split dentro de un único dataset no debe confundirse con validación externa; en muestras pequeñas es una forma débil e ineficiente de validación interna.
- **Steyerberg y Harrell:** el bootstrap y el remuestreo interno suelen ser preferibles al split-sample cuando el N es limitado.
- **Arlot & Celisse:** la validación cruzada es una herramienta útil para estimación interna de riesgo y selección/model checking.
- **Hernán & Robins:** en estudios observacionales, la asociación no debe presentarse como causalidad sin un diseño causal explícito.
- **STROBE:** el reporte final debe explicitar diseño, muestra, variables, missingness, supuestos y limitaciones.

Estas referencias deben quedar registradas en `mvp_decisions.yaml`, `FASE5_CIERRE_TECNICO.md` y, si corresponde, en el README de Fase 5.

### 6.5 Implementación sugerida

```python
AI_LEADERS = {"USA", "CHN", "SGP", "ARE", "IRL", "ISR", "KOR", "JPN"}
LARGE_AI_POWERS = {"USA", "CHN", "IND", "JPN"}
LATAM_PEERS = {"ARG", "BRA", "CHL", "COL", "CRI", "MEX", "PER", "URY"}
EU_LAGGARDS = {"GRC", "ROU", "HRV"}

FORBIDDEN_MEMBERSHIP_COLUMNS = {
    "split", "train", "test", "holdout", "fold", "partition", "set"
}


def build_analysis_sample_membership(feature_matrix: pd.DataFrame) -> pd.DataFrame:
    required = ["iso3", "country_name_canonical"]
    for col in required:
        if col not in feature_matrix.columns:
            raise ValueError(f"Missing required column for membership: {col}")

    optional = ["region", "income_group"]
    cols = required + [c for c in optional if c in feature_matrix.columns]
    membership = feature_matrix[cols].copy()

    membership["is_primary_analysis_sample"] = True
    membership["sample_inclusion_category"] = "mvp_preregistered"
    membership["inclusion_reason"] = "Included in preregistered 43-country MVP sample"
    membership["is_chile_focal"] = membership["iso3"].eq("CHL")
    membership["is_ai_leader_sensitivity"] = membership["iso3"].isin(AI_LEADERS)
    membership["is_large_ai_power_sensitivity"] = membership["iso3"].isin(LARGE_AI_POWERS)
    membership["is_latam_peer_sensitivity"] = membership["iso3"].isin(LATAM_PEERS)
    membership["is_eu_laggard_sensitivity"] = membership["iso3"].isin(EU_LAGGARDS)
    membership["is_eu_member_or_laggard_sensitivity"] = membership["is_eu_laggard_sensitivity"]

    if "region" not in membership.columns:
        membership["region"] = pd.NA
    if "income_group" not in membership.columns:
        membership["income_group"] = pd.NA

    membership["has_comparable_region_income"] = membership["region"].notna() & membership["income_group"].notna()
    membership["leave_group_region"] = membership["region"]
    membership["leave_group_income"] = membership["income_group"]
    membership["special_case_flag"] = "none"
    membership.loc[~membership["has_comparable_region_income"], "special_case_flag"] = "metadata_incomplete"
    membership["notes"] = "Primary analysis sample; sensitivity flags only; no train/test split."

    forbidden = FORBIDDEN_MEMBERSHIP_COLUMNS.intersection(set(membership.columns))
    if forbidden:
        raise ValueError(f"Forbidden membership columns present: {sorted(forbidden)}")

    if len(membership) != 43:
        raise ValueError(f"Expected 43 countries, got {len(membership)}")
    if membership["iso3"].nunique() != 43:
        raise ValueError("Expected 43 unique ISO3 codes")
    if "CHL" not in set(membership["iso3"]):
        raise ValueError("CHL must be present in analysis sample membership")

    return membership
```

## 7. Política de transformaciones no estimables

### 7.1 Problema

El workbook actual evidencia transformaciones marcadas como `zero_mad_or_not_estimable`. Esto ocurre cuando una variable no tiene variabilidad suficiente o el MAD es cero, lo que impide calcular correctamente un z-score robusto.

El error a evitar es que estas columnas derivadas entren silenciosamente en Fase 6 como si fueran features válidas.

### 7.2 Regla metodológica

Las variables raw deben preservarse. Las columnas derivadas no estimables deben:

1. mantenerse documentadas en `mvp_transform_params.csv`;
2. marcarse con status explícito;
3. excluirse de los grupos de modelado primario de Fase 6;
4. aparecer en el workbook auditable como transformaciones no utilizables para estimación primaria;
5. reportarse en `fase5_manifest.json`.

### 7.3 Estados permitidos de transformación

```text
ok
not_applicable
zero_mad_or_not_estimable
constant_or_quasi_constant
insufficient_non_missing_values
source_missing
```

### 7.4 Columnas mínimas en `mvp_transform_params.csv`

```text
variable_original
variable_derived
transform_type
n_non_missing
median
mad
mean
std
min
max
status
used_in_primary_modeling
exclusion_reason
notes
```

### 7.5 Regla exacta para `used_in_primary_modeling`

```python
NON_ESTIMABLE_STATUSES = {
    "zero_mad_or_not_estimable",
    "constant_or_quasi_constant",

    transform_params["status"].isin(NON_ESTIMABLE_STATUSES),
```

## 19. Resumen ejecutivo del cambio

La nueva Fase 5 debe ser el cerrojo metodológico del proyecto. Su función no es modelar, predecir ni reservar países para prueba. Su función es dejar una base auditable y contractual que proteja la credibilidad de todo el pipeline.

La corrección mínima no basta. No es suficiente borrar la columna `split` del CSV. Hay que eliminar el split del código, del bundle, del API, de los tests, del Excel, del diccionario, de los manifiestos y de la narrativa. Además, hay que agregar dos mejoras indispensables detectadas en la investigación final:

1. una política explícita para transformaciones no estimables;
2. una limpieza documental completa del workbook auditable.

Solo después de esto Fase 6 podrá operar correctamente como fase de estimación de asociaciones ajustadas y no como una falsa evaluación predictiva.
